# Brief: plant-image-management

## Problem
観葉植物初心者は、植物の成長を写真で残したいが、現在の画像扱いは植物詳細体験として整理されていない。画像が植物個体の成長記録として時系列に見返せず、一覧で植物を見分ける代表画像の設定や、画像枚数上限の把握もできない。

## Current State
`plant-photo-schema-foundation` により、植物個体に複数の写真記録を紐づけ、代表写真を nullable に参照するための基盤仕様は存在する。ただし、既存 spec は画像アップロード、画像ファイル保存・削除、写真 CRUD API、成長記録ギャラリー UI を明示的に対象外としている。

## Desired Outcome
ユーザーは植物詳細画面から、現在表示している植物に対して画像をアップロードできる。植物詳細ではその植物に紐づく画像だけが時系列ギャラリーとして表示され、現在の画像枚数と上限を把握できる。ユーザーはギャラリー内の画像から代表画像を設定でき、植物一覧ではその代表画像がサムネイルとして表示される。画像を削除するときは確認でき、代表画像を削除した場合は代表画像が未設定に戻る。

## Approach
植物詳細統合型の画像管理として実装する。画像アップロード対象を現在表示中の植物に限定し、他植物への紐付け変更は提供しない。既存の写真記録・代表写真基盤を利用し、Frontend は植物詳細画面の一部としてギャラリーと操作を構成し、Backend は owner-scoped な植物写真 API と上限判定を提供する。

この approach は、画像を独立した管理対象ではなく植物との生活記録として扱う Green Mate のプロダクト方針と一致する。また、将来の成長記録の振り返りやタイムラプス表示にもつながる時系列データ構造を保てる。

## Scope
- **In**: 植物詳細画面からの画像アップロード、現在表示中の植物への画像紐付け、植物ごとの時系列ギャラリー、現在枚数と上限の表示、上限なしユーザーの上限表示省略、ギャラリー内画像からの代表画像設定、植物一覧での代表画像表示、画像削除、削除確認ダイアログ、代表画像削除時の明示と未設定化
- **Out**: 他植物への画像紐付け変更、画像管理専用画面、プラン定義、課金、画像編集、トリミング、サムネイル生成、画像変換、公開共有、タイムラプス表示、植物種マスタとの連携

## Boundary Candidates
- 植物写真 API: 現在ユーザーの owner scope と対象植物の所有確認を行い、写真の作成・一覧・削除・代表画像設定を扱う。
- 植物詳細 UI: 詳細画面の一部としてアップロード導線、時系列ギャラリー、上限表示、代表画像設定、削除確認を扱う。
- 植物一覧 UI: 既存の植物一覧表示に代表画像サムネイルを表示し、代表画像未設定時の空状態を扱う。
- ユーザー画像上限: 一般ユーザーの1植物5枚制限と、特定ユーザーの上限なし判定を domain validation として扱う。

## Out of Boundary
- 画像ファイルの高度な最適化やサムネイル生成。
- 画像を他植物へ移動または再紐付けする操作。
- ユーザー向けのプラン管理、課金、管理画面。
- タイムラプス表示そのもの。
- 公開ギャラリー、共有、SNS 投稿。

## Upstream / Downstream
- **Upstream**: `auth-authorization-foundation` の internal owner id と保護 API ルール、`plant-photo-schema-foundation` の写真記録・代表写真参照、既存の植物一覧・詳細 API と画面。
- **Downstream**: 成長記録の振り返り強化、タイムラプス表示、写真コメントや撮影日の活用、画像ストレージ・配信方式の改善。

## Existing Spec Touchpoints
- **Extends**: `plant-photo-schema-foundation`。画像上限判定、特定ユーザーの上限なし判定に必要な user column、代表画像削除時の未設定化が既存基盤の前提と整合するか確認する。
- **Adjacent**: `plant-registration` は植物個体の作成を扱うが、本 spec は登録時画像ではなく詳細画面からの成長画像追加を扱う。`plant-profile-editing` は植物基本情報の編集を扱うが、画像ギャラリー管理や代表画像設定は本 spec に置く。`plant-watering-care` はお世話記録を扱うが、成長写真の時系列表示とは責務を分ける。

## Constraints
Frontend は Vue 3、Vite、TypeScript、Tailwind CSS、Vue Router の既存構成に従う。Backend は FastAPI、Pydantic、SQLModel、SQLAlchemy Session、Alembic の layered architecture に従う。ユーザー所有データは `CurrentUser.id` による owner scope を必須にし、owner id を request body や response に露出しない。一般ユーザーの画像上限は定数で定義し、特定ユーザーの上限なし判定は当面 DB の値を直接変更して運用できる形にする。
