# 実装計画

- [ ] 1. 植物基本情報更新 API の基盤を追加する
  - [ ] 1.1 既存植物項目だけを受け取る更新 contract と検証を追加する
    - 植物名、家に来た日、メモ、水やり周期だけを更新入力として扱う。
    - 植物名必須、水やり周期の正の整数、不正日付の rejection を登録時の意味と揃える。
    - 既存テーブルにない植物属性や owner 情報は更新対象にしない。
    - 完了条件: 有効な更新入力は正規化され、無効な更新入力は 422 相当の検証エラーとして扱われる。
    - _Requirements: 2.1, 2.2, 2.3, 2.4, 3.1, 3.2, 3.3, 3.4, 3.6, 6.5_
    - _Boundary: Plant schema, PlantService_

  - [ ] 1.2 owner-scoped な植物更新の永続化を追加する
    - 認証済み owner id と植物 id の両方に一致する植物だけを更新する。
    - 更新対象は既存の基本情報に限定し、owner、cover photo、最新水やり日時、水やり記録は変更しない。
    - 更新成功時に更新日時が変わり、他 owner の植物は見つからない扱いになる。
    - 完了条件: 自分の植物だけが更新され、他ユーザーの植物 row は変更されない。
    - _Requirements: 4.1, 6.1, 6.3, 6.4, 6.5, 7.1, 7.3, 7.4_
    - _Boundary: PlantRepository, PlantService_

  - [ ] 1.3 植物更新 endpoint を公開する
    - 認証済みユーザーだけが植物基本情報の更新を実行できる。
    - 検証エラー、認証エラー、無効ユーザー、存在しない植物、他 owner の植物を既存方針どおりの応答に変換する。
    - 更新成功時は owner 情報を含まない更新後の植物詳細を返す。
    - 完了条件: `PATCH /plants/{plant_id}` が自分の植物だけを更新し、更新後の `PlantRead` を返す。
    - _Requirements: 4.1, 4.5, 5.1, 6.1, 6.2, 6.3, 6.4, 6.5_
    - _Boundary: PlantsRouter_

- [ ] 2. フロントエンドの更新入力と状態管理を追加する
  - [ ] 2.1 (P) 植物更新の型と API 呼び出しを追加する
    - 既存植物項目だけを表す更新入力型を追加する。
    - authenticated API client 経由で植物更新を呼び出せるようにする。
    - owner 情報や既存テーブルにない植物属性を型や request payload に追加しない。
    - 完了条件: フロントエンドから型付きで植物更新 API を呼び出せる。
    - _Requirements: 4.1, 6.2, 6.4, 6.5_
    - _Boundary: Frontend plant types, plants API client_

  - [ ] 2.2 (P) 登録と編集で共有できる入力検証を追加する
    - 植物名、家に来た日、メモ、水やり周期の正規化と検証を共通化する。
    - 任意項目の空文字を未入力として扱い、入力エラー時は修正可能な状態を返す。
    - 完了条件: 編集フォームが API 送信前に植物名、日付、水やり周期の入力エラーを判定できる。
    - _Requirements: 2.3, 3.1, 3.2, 3.3, 3.4, 3.5, 3.6_
    - _Boundary: plantFormValidation_

  - [ ] 2.3 植物詳細の更新 state を composable に追加する
    - 保存中、保存成功、保存失敗の状態を詳細画面から扱えるようにする。
    - 更新成功時は取得済みの植物 state を更新後の植物に差し替える。
    - 更新失敗時は既存の植物詳細を残し、入力見直しや再試行に戻れる状態にする。
    - 完了条件: 詳細画面から更新処理を呼ぶと、成功時は plant state が更新され、失敗時は update error が残る。
    - _Depends: 2.1, 2.2_
    - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5, 5.1, 6.2_
    - _Boundary: usePlantDetail_

- [ ] 3. 詳細画面の編集 UI を実装して統合する
  - [ ] 3.1 (P) 植物詳細に編集導線を追加する
    - 植物詳細が取得できている時だけ編集導線を表示する。
    - 詳細取得失敗中は編集を完了可能な操作として表示しない。
    - 既存項目の未入力表示を維持する。
    - 完了条件: 植物詳細で編集を開始でき、取得失敗時には編集開始できない。
    - _Requirements: 1.1, 1.5, 5.1, 5.2, 5.3, 5.4, 5.6, 8.1, 8.2, 8.3_
    - _Boundary: PlantDetail_

  - [ ] 3.2 (P) 既存項目だけを編集するフォームを追加する
    - 現在の植物名、家に来た日、メモ、水やり周期を初期値として表示する。
    - 保存中は重複保存を防ぎ、取り消し時は未保存の変更を捨てる。
    - エラー表示後も入力内容を見直せるようにする。
    - 完了条件: 編集フォームで既存項目を変更、保存、取り消しでき、小さな画面でも主要操作が読み取れる。
    - _Requirements: 1.2, 1.3, 1.4, 2.1, 2.3, 3.5, 4.2, 4.4, 7.5, 8.1, 8.2, 8.3, 8.4_
    - _Boundary: PlantEditForm_

  - [ ] 3.3 詳細画面に編集フローを統合する
    - 詳細表示と編集フォームを画面内で切り替える。
    - 保存成功後に詳細表示へ戻り、成功メッセージと更新後の植物情報を表示する。
    - 水やり周期が変わった場合は水やり状態を再読込し、既存の水やり記録は変更しない。
    - 完了条件: ユーザーが詳細画面で編集、保存、取り消しを行い、保存後に更新後の詳細と水やり目安を確認できる。
    - _Depends: 2.3, 3.1, 3.2_
    - _Requirements: 1.1, 1.2, 1.4, 4.1, 4.2, 4.3, 4.4, 5.1, 5.2, 5.3, 5.4, 5.5, 5.6, 7.1, 7.2, 7.3, 8.5_
    - _Boundary: PlantDetailPage, usePlantWatering integration_

- [ ] 4. 更新機能の回帰テストとスモーク検証を追加する
  - [ ] 4.1 バックエンド更新 API の正常系と保護境界をテストする
    - 更新成功、任意項目のクリア、検証エラー、他 owner 404、owner field 非公開を検証する。
    - 更新失敗時に対象 row が変更されないことを確認する。
    - 既存テーブルにない植物属性や DB migration が追加されていないことを確認する。
    - 完了条件: backend の植物更新関連テストが成功し、他 owner の植物は更新できない。
    - _Depends: 1.3_
    - _Requirements: 2.1, 2.2, 2.3, 3.1, 3.2, 3.3, 3.4, 4.1, 4.5, 6.1, 6.2, 6.3, 6.4, 6.5, 7.1, 7.3, 7.4_

    - 更新後の植物名が水やり実績表示で現在名として参照されることを確認する。
    - ownerless plant が作られず、他 owner の植物が更新されないことを確認する。
    - 完了条件: local SQLite と Turso 向け smoke path で、作成、更新、水やり、読み取りの一連の検証が成功する。
    - _Depends: 1.3_
    - _Requirements: 5.1, 5.2, 5.5, 6.1, 6.3, 6.4, 7.1, 7.2_
    - _Boundary: Smoke verification_

  - [ ] 4.3 フロントエンドビルドと主要 UI フローを検証する

    - 完了条件: frontend build が成功し、詳細画面の編集フローが主要状態で操作できる。
